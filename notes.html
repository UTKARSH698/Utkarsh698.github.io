The AWS_ENDPOINT_URL seam — or, why my Grafana Alloy PR took twice as long as I expected.

Notes from shipping PR #147 to grafana/alloy-scenarios — a CloudWatch metrics scenario that runs without a real AWS account. The Alloy config wasn’t the hard part. The hard part was a seam between two layers of AWS’s own SDK that nothing in the Alloy docs warns you about.

The scenario is small on paper. LocalStack on port 4566 pretending to be CloudWatch and STS. A Python boto3 seeder pushing AWS/EC2 CPUUtilization metrics every 30 seconds so there’s a signal to scrape. Grafana Alloy configured with prometheus.exporter.cloudwatch, pointed at LocalStack instead of real AWS. Prometheus scrapes Alloy. End-to-end metrics visible in ~90 seconds.

Where it broke

Alloy’s CloudWatch exporter is a thin wrapper over YACE (Yet Another CloudWatch Exporter). YACE in turn talks to AWS through the standard AWS SDK for Go. Neither YACE nor the Alloy block exposes an endpoint-url flag the way the AWS CLI does. The block accepts a region and a set of credentials and then trusts the SDK to do the right thing.

The trick — undocumented in Alloy’s reference, but baked into the SDK since v1 — is that the AWS Go SDK reads an environment variable called AWS_ENDPOINT_URL and routes every outgoing API call through that endpoint when set. Pass it through the container environment and suddenly the YACE-shaped Alloy block is happily querying LocalStack, with no patches to YACE and no fork of Alloy.

The seeder needs the same variable for the same reason: boto3 reads AWS_ENDPOINT_URL too. If you set it on one container and forget it on the other, you get a quiet half-broken system — LocalStack receives no metrics, but Alloy queries succeed (with an empty result set), and Prometheus shows healthy scrapes with zero series. The failure looks like a config typo, but it’s actually two services living in two different versions of reality.

What the scenario does about it

One variable, declared once at the top of docker-compose.yml, threaded into both the Alloy container and the seeder container. A short paragraph in the README explaining why — because the next contributor will hit this exact seam, and there’s no point making them re-derive it.

The whole PR is six small files. The interesting work was an hour of reading SDK source to confirm AWS_ENDPOINT_URL is the right hook (as opposed to something more invasive like a custom HTTP client) and another hour verifying that both YACE-via-Alloy and boto3 honour it identically. Merged by Grafana DevRel with all 12 CI checks green: PR #147.

The lesson, generalised

When a thing "just needs an endpoint override," check whether the underlying SDK already has an override mechanism before you reach for a wrapper, a fork, or a patch. The AWS SDKs in particular have at least three layers of configuration (env var → shared config → per-call client option) and they tend to merge in surprising ways. The cheapest fix is almost always the lowest one in that stack that does what you need — anything higher up is a code change someone has to maintain.

The scenario is now live at grafana/alloy-scenarios/cloudwatch-metrics — clone it and you’ll have a working CloudWatch-to-Prometheus pipeline running locally in roughly the time it takes to read this post.
