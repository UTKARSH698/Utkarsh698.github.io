The AWS_ENDPOINT_URL seam — or, why my Grafana Alloy PR took twice as long as I expected.

Notes from shipping PR #147 to grafana/alloy-scenarios — a CloudWatch metrics scenario that runs without a real AWS account. The Alloy config wasn’t the hard part. The hard part was a seam between two layers of AWS’s own SDK that nothing in the Alloy docs warns you about.

The scenario is small on paper. LocalStack on port 4566 pretending to be CloudWatch and STS. A Python boto3 seeder pushing AWS/EC2 CPUUtilization metrics every 30 seconds so there’s a signal to scrape. Grafana Alloy configured with prometheus.exporter.cloudwatch, pointed at LocalStack instead of real AWS. Prometheus scrapes Alloy. End-to-end metrics visible in ~90 seconds.

Where it broke

Alloy’s CloudWatch exporter is a thin wrapper over YACE (Yet Another CloudWatch Exporter). YACE in turn talks to AWS through the standard AWS SDK for Go. Neither YACE nor the Alloy block exposes an endpoint-url flag the way the AWS CLI does. The block accepts a region and a set of credentials and then trusts the SDK to do the right thing.

The trick — undocumented in Alloy’s reference, but baked into the SDK since v1 — is that the AWS Go SDK reads an environment variable called AWS_ENDPOINT_URL and routes every outgoing API call through that endpoint when set. Pass it through the container environment and suddenly the YACE-shaped Alloy block is happily querying LocalStack, with no patches to YACE and no fork of Alloy.

The seeder needs the same variable for the same reason: boto3 reads AWS_ENDPOINT_URL too. If you set it on one container and forget it on the other, you get a quiet half-broken system — LocalStack receives no metrics, but Alloy queries succeed (with an empty result set), and Prometheus shows healthy scrapes with zero series. The failure looks like a config typo, but it’s actually two services living in two different versions of reality.

What the scenario does about it

One variable, declared once at the top of docker-compose.yml, threaded into both the Alloy container and the seeder container. A short paragraph in the README explaining why — because the next contributor will hit this exact seam, and there’s no point making them re-derive it.

The whole PR is six small files. The interesting work was an hour of reading SDK source to confirm AWS_ENDPOINT_URL is the right hook (as opposed to something more invasive like a custom HTTP client) and another hour verifying that both YACE-via-Alloy and boto3 honour it identically. Merged by Grafana DevRel with all 12 CI checks green: PR #147.

The lesson, generalised

When a thing "just needs an endpoint override," check whether the underlying SDK already has an override mechanism before you reach for a wrapper, a fork, or a patch. The AWS SDKs in particular have at least three layers of configuration (env var → shared config → per-call client option) and they tend to merge in surprising ways. The cheapest fix is almost always the lowest one in that stack that does what you need — anything higher up is a code change someone has to maintain.

The scenario is now live at grafana/alloy-scenarios/cloudwatch-metrics — clone it and you’ll have a working CloudWatch-to-Prometheus pipeline running locally in roughly the time it takes to read this post.

Three RBAC invariants that hold per-service and break under concurrency — or, why local correctness doesn’t compose.

While building my own AgriFuture marketplace prototype I hit a concurrent-order race that smelled like an authorization bug. So I formalised it. This is the short version of an independent preprint (Zenodo · DOI 10.5281/zenodo.20686317): three role-based access-control invariants that each pass when you check a single service, and each fail the moment two services touch shared state at the same time.

The setup is the boring kind of distributed system everyone actually runs: an identity service that owns roles and permissions, an API gateway that caches policy decisions, and an order service that runs transactions. Each one is tested. Each one is "correct." And yet a permission a user no longer holds can still authorise an action — not because any single service is wrong, but because correctness checked in isolation says nothing about correctness under interleaving.

The three invariants

I picked three properties that any RBAC system would claim to guarantee:

RPS — Role-Permission Separation. A cached authorization decision must never grant a permission the user’s current roles don’t confer. TAR — Transaction Atomicity under Revocation. If a role is revoked mid-transaction, the transaction must not commit using the revoked authority. APC — API Gateway Policy Consistency. The gateway’s cached policy must agree with the identity service’s source of truth.

The subtlety is in how you state RPS. The obvious form — "the cache for a role is a subset of that role’s permissions" — is trivially true and tells you nothing. The form that actually bites is indexed by user: cache[s][u] ⊆ permissions(roles[u]). Once the invariant is about the user’s current roles, a revocation that lands between "gateway reads cache" and "identity service updates roles" breaks it.

What the model checker found

I wrote two specifications — a buggy design and a fixed one — and walked the complete reachable state space with a breadth-first checker (TLC-equivalent: it explores every interleaving, not a sample). The numbers are the whole point, so here they are exactly:

The buggy specification reaches 188,160 distinct states, and all three invariants are violated — each one has a concrete counterexample trace: a specific interleaving of revoke / cache-read / commit that ends in an unauthorised action. The fixed specification — which makes the revoke-and-invalidate step atomic with respect to the cache — collapses to 4,928 states and holds all three clean. A 38× smaller reachable space, and zero violations.

The collapse in state count is itself the lesson: the buggy version is "bigger" because it admits a huge number of inconsistent intermediate worlds that the fixed version simply makes unreachable.

The honest scope

This is a preprint, not peer-reviewed, and it’s independent work — done by me, on a problem I ran into myself, not a supervised thesis. The model is a deliberately small abstraction of a real system, which is exactly what makes exhaustive checking tractable; it is not a claim that any production RBAC implementation has these specific bugs. What it does show, rigorously, is that per-component verification is not closed under composition — and that the gap is reachable, not theoretical.

Spec, checker, traces, and the full write-up are open: github.com/UTKARSH698/security-invariants-rbac. Clone it, run the checker, and you’ll reproduce both numbers yourself.
