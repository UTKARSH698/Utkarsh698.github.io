Feature № 01 · Distributed Systems Shipped · Mar 2026
01

CloudFlow — a SAGA-orchestrated order pipeline.

A checkout system that never double-charges or oversells — even when a payment provider fails mid-order. Built on AWS as a Step Functions SAGA with compensation, an event-sourced order log in DynamoDB, idempotent replay-safe handlers, and a DynamoDB-backed circuit breaker that survives Lambda cold starts. Sustained 1,100+ req/min at p50 47 ms.

E-commerce checkout is the textbook example of a workflow where things go wrong in interesting ways. Inventory locks, payment processors return 5xx, and you still need to either ship the order or release the stock. CloudFlow models the entire flow as a SAGA — a sequence of forward steps each paired with a compensating action, orchestrated by AWS Step Functions, with every state transition appended to an immutable event log in DynamoDB.

i. The problem worth solving

Naive synchronous checkout pipelines hide failure: when one step times out, the others have already mutated state. The system needs atomicity across services that don’t share a transaction. SAGA gives you that — but only if compensations are idempotent, retries are safe, and partial failure of the orchestrator itself doesn’t leave money on the table.

ii. Architecture

iii. The hard part

The interesting failure mode wasn’t payment outages — it was oversell under concurrent decrement. Two orders hitting inventory at the same time can both pass a check-and-decrement if the read isn’t atomic. I solved it with DynamoDB conditional writes (UpdateItem with a quantity >= requested condition); one writer wins, the other catches ConditionalCheckFailedException and the SAGA compensates cleanly. The same primitive does double duty for idempotency: every handler uses an attribute_not_exists conditional insert keyed by the request’s idempotency token, which closes the at-least-once SQS replay vector at the database layer rather than in application code.

Under load test, the circuit breaker fast-fails in under a millisecond when the payment processor is degraded, instead of letting Step Functions burn 30 seconds waiting for a provider timeout. — Notes from scripts/load_test.py against LocalStack

iv. Measured behaviour

Against LocalStack, 10 concurrent threads / 50 orders gave 1,100+ req/min, P50 47 ms, P95 89 ms, P99 120 ms. Throughput plateaus near 1,300 req/min at 20 threads — the LocalStack Docker container is the bottleneck, not the design. On real AWS, DynamoDB partitions scale horizontally and the same correctness properties hold; only the absolute clock changes. 48 tests cover failure paths under both moto mocks (unit) and LocalStack (integration), including duplicate orders, payment declines, circuit-open fast-fail, and atomic oversell rejection.

v. The open consistency question

Compensation is designed so that partial rollback cannot produce an exploitable inconsistent state, but that property is argued by case analysis over the failure modes I could construct — not proved. If Step Functions itself fails mid-compensation, the system relies on the assumption that every retried compensating step eventually delivers, which the AWS SLA does not guarantee under all disruption modes. This is the same shape of gap that shows up in CSPM remediation (per-check fixes that don’t compose into system-level safety) and in CloudPulse (speed-path / batch-path divergence) — the throughline of the research I want to pursue: formal verification of distributed security and consistency policies, not just per-component tests.

Feature № 02 · Real-Time Analytics Shipped · Mar 2026
02

CloudPulse — a real-time analytics platform on AWS serverless.

Live dashboards that update in ~12 ms, while the same event stream still answers 90-day historical queries (~1.8 s). A production-grade Lambda Architecture — batch and speed layers over one stream, Kinesis dual-write at ingest, Hive-partitioned S3 + Athena for history, DynamoDB counters for the live view. Five Lambdas, 15 Terraform files, inside the AWS Free Tier.

Most analytics tutorials end at "send events to a database, query later." The interesting failures live in the seam between real-time and historical — the dashboard that updates instantly versus the SQL that scans ninety days. CloudPulse is built on the Lambda Architecture pattern: two paths, one source of truth, an honest answer to what should I trust right now?

i. The pattern

Every event hits a single Cognito-authed ingest endpoint and forks. The batch path lands in S3 in a Hive-partitioned layout (events/year=…/event_type=…/) and becomes queryable via Athena SQL — slow, durable, complete. The speed path lands in DynamoDB as per-minute counters with 24-hour TTL and is queryable in ~12 ms — fast, lossy, recent. Same data, two materializations, chosen at query time.

ii. Architecture

iii. The dual-write decision

The ingest Lambda writes to SQS and Kinesis on every request. SQS is the durable enqueue — if anything else breaks, S3 still ends up with the event. Kinesis is fail-open — if it errors, the live counters lag for a minute but the historical record is never at risk. That asymmetry is deliberate. The dashboard going stale is an inconvenience; losing the audit trail is a different category of problem.

Speed answers in 12 ms. Batch answers in ~1.8 s after partition pruning. The dual-write costs ~30 ms at ingest. Everything fits inside the AWS Free Tier. — Dev deployment benchmarks

iv. The open consistency question

The two layers can diverge. Speed says 4,200 events this minute; batch (a few minutes later) says 4,215. Most of the time it doesn't matter. But if anyone uses the speed layer for anomaly detection — error-rate alerting, fraud-spike triggers — a diverged counter could suppress a real signal until the batch path corrects it, possibly after the incident window has closed. That's the same problem CSPM and CloudFlow surface in different clothing: per-component correctness doesn't compose into system-level correctness. It's the research problem, restated.

Feature № 03 · Cloud Security Shipped · Feb 2026
03

CSPM — automated posture management for AWS.

Finds the misconfigurations behind most cloud breaches — public S3 buckets, over-permissive security groups, risky IAM — and safely auto-fixes the ones it can. A serverless agent running 23 CIS-v1.5 checks across S3, IAM, EC2 Security Groups, and CloudTrail; scores compliance, alerts on critical findings, DRY_RUN-by-default until you flip the flag.

Cloud breaches almost never come from sophisticated attacks. They come from an S3 bucket left public, SSH open to the world, a root account with no MFA. CSPM catches the boring ones automatically, alerts you the moment they appear, and fixes the safe ones without human intervention. The interesting design choices live in which findings it’s willing to fix without asking — and what it does about the ones it isn’t.

i. What it checks

Twenty-three checks against the CIS AWS Foundations Benchmark v1.5, grouped across four services: four S3 checks (Block Public Access, versioning, access logging, default encryption); four IAM checks (root MFA, root access keys, password policy, access-key age > 90 days); nine EC2 Security Group checks (SSH, RDP, MySQL, PostgreSQL, MongoDB, Redis, Elasticsearch, all-traffic, default SG); and six CloudTrail checks (trail exists, multi-region, log-file validation, CloudWatch Logs integration, log-bucket privacy, logging state). Each rule emits a finding with severity, asset reference, and an explicit remediation step.

ii. Two details most tools miss

Security-group checks usually match on the exact port number. A rule allowing TCP 0–65535 exposes SSH but evades a tool that only looks for port=22. CSPM’s SG checks are port-range aware — the sensitive port is flagged if it falls anywhere within the rule’s allowed range. Similarly, most tools only cover IPv4 (0.0.0.0/0) and skip IPv6 (::/0); CSPM checks both, independently.

iii. Architecture

iv. DRY_RUN by default

The remediator ships with DRY_RUN=true. It logs every fix it would make without touching anything. You flip the flag once you’ve reviewed your first findings. Even with auto-remediation enabled, only safe actions are taken: enabling S3 Block Public Access, enabling S3 versioning, and revoking open SSH / RDP / all-traffic Security Group rules. IAM and CloudTrail findings are flagged for human review — automatically rotating access keys or modifying trail config in production is a category of risk this system deliberately refuses.

First scan on a brand-new AWS account: 80% compliance, three critical findings. Scanner finishes in under six seconds regardless of finding count — the remediator runs async via InvocationType=Event. — Initial deployment log

v. The open formal question

Applying fix A then fix B in dependency order is locally safe. What it isn’t is formally proved complete under all concurrent misconfiguration states. Under k simultaneous misconfigurations with interdependent remediation actions, does dependency-ordering always produce a globally consistent final state? The current heuristic prevents the cases I could construct — but I cannot prove it handles all of them. That’s the same gap CloudFlow and CloudPulse surface in different clothing: per-component correctness doesn’t compose into system-level correctness. The research problem, again.

Feature № 04 · Open Source · Observability Merged · May 2026
04

Grafana Alloy — a CloudWatch metrics scenario, merged upstream.

PR #147 to grafana/alloy-scenarios — a self-contained, AWS-free walkthrough of pulling CloudWatch metrics into Prometheus through Grafana Alloy. Closed issue #98; merged by Grafana DevRel with all 12 CI checks green.

Grafana Alloy is Grafana’s OpenTelemetry-flavoured collector. Its alloy-scenarios repo is the official examples bench — each folder is a runnable docker-compose stack a new user can clone and see real telemetry inside two minutes. Issue #98 asked for a CloudWatch metrics scenario that didn’t require an AWS account, so reviewers and curious engineers could try it without spinning up real infrastructure. I picked it up and shipped the scenario end-to-end.

i. The architecture

Six files, one docker compose up. LocalStack emulates the CloudWatch and STS APIs on port 4566. A Python boto3 seeder publishes synthetic AWS/EC2 CPUUtilization metrics on a 30-second cadence so the scenario has a live signal to scrape. Alloy’s prometheus.exporter.cloudwatch block is pointed at LocalStack via AWS_ENDPOINT_URL; the YACE SDK underneath doesn’t care that the endpoint isn’t real AWS. Prometheus scrapes Alloy, and metrics appear in the local stack within ~90 seconds of startup.

ii. The part that took the longest

The interesting bit wasn’t the Alloy config — it was the seam where Alloy’s exporter trusts the AWS SDK to do the right thing. The exporter doesn’t expose an endpoint-url knob; it expects an SDK already configured to hit real AWS. Threading AWS_ENDPOINT_URL through the container environment so both the exporter and the seeder agree on where "AWS" lives took longer to verify than to write. That detail is now documented in the scenario README so the next contributor doesn’t re-derive it.

iii. What it’s for

New engineers learning Alloy can clone the scenario and have a working CloudWatch-to-Prometheus pipeline running locally before lunch. Existing Alloy users have a known-good config to copy when they wire up real AWS. The PR also pinned a LocalStack version (image-versions.env) so the scenario doesn’t silently rot when upstream containers update.

Huge thank you for this contribution. — Jayclifford345, Grafana DevRel, on merge

Feature № 05 · AI Platform · Final-Year Project Live on Render · 2025
05

AgriFuture — six Gemini AI modules behind one Express server.

An AI-powered agricultural intelligence platform: crop recommendation, disease detection, drone terrain analysis, market forecasting, digital twin simulation, and conversational AgriBot — each backed by Google Gemini with a different prompting strategy. Plus live mandi prices, Razorpay commerce with HMAC verification, JWT auth with email + SMS OTP, and a PWA that works offline.

AgriFuture started as a final-year project and ended up being the one I built end-to-end myself — 56 TypeScript files, ~12,500 lines, six AI modules, real payments, real OTP, and a PWA shell. It’s the project where I learned that “put an LLM behind it” isn’t the engineering — the engineering is everything around the LLM that makes it safe to expose.

i. Six AI modules, one Gemini key

A single Express 5 server routes to six distinct AI surfaces via geminiService.ts: Crop Recommendation (soil parameters → structured JSON with confidence scores), Disease Detector (multi-modal: image + symptom text → severity Low/Moderate/High + pathogen + treatment), AgriDrone V2 (coordinate → satellite terrain description → NDVI / irrigation map / soil-health gauge), Market Forecasting (30-day price predictions with sentiment), Digital Twin (visual node-network field simulation), and AgriBot (streaming conversational AI). Crop / Disease / Drone / Market sit on Gemini 2.5 Flash; Digital Twin and AgriBot use Gemini 3 Flash. Each module is a different prompting strategy — structured JSON extraction, multi-modal vision, coordinate-grounded narrative — not the same prompt with different inputs.

ii. Security is middleware, not afterthought

The interesting engineering on a project like this isn’t the AI calls — it’s the boundary around them. JWT in httpOnly cookies (not localStorage), bcrypt with 10 salt rounds, OTP via both Nodemailer (Gmail SMTP) and MSG91 SMS with cryptographically random codes stored in Postgres with a 15-minute TTL so server restarts don’t invalidate them, rate limiting at the middleware layer (5 OTP / 15 min and 10 login / 15 min per IP), Helmet security headers, and Zod schema validation on every input — the same schemas run on the client and the server, so the API can’t receive a shape it didn’t validate. Razorpay payments are committed only after server-side HMAC signature verification, which closes the forged-completion-event replay vector that breaks naive checkout flows.

The same Gemini API key routes to six distinct intelligence layers via one dispatcher — what looks like “an AI platform” from outside is really six different prompt-engineering decisions stitched behind one auth and rate-limit boundary. — Design note

iii. Surface

React 19 + TypeScript 5.8 + Vite 6 + Tailwind 4 + Framer Motion 12 on the client; Express 5 + Node.js + PostgreSQL on the server; 44+ shared TypeScript interfaces between client and server so the same Zod schemas validate at the form layer and the API boundary. Live mandi prices (27+ crops) and Open-Meteo weather (no API key) are hydrated server-side so the client never waits on a third-party. AgriStore (58+ products) wraps Razorpay with HMAC verification; an admin dashboard with bar charts and sparklines surfaces user activity, expense tracking, and report history. PWA with cache-first service worker, install prompts, and seven Indian-language localisations. CI/CD via GitHub Actions; one-click deploys configured for both Render (render.yaml) and Railway (railway.json).

iv. Honest limits

Gemini API latency adds 1–3 s per AI call and there’s no response-caching layer — tolerable for an advisory product, not for high-traffic. Single-region Postgres on Render free tier with no read replicas or pgBouncer pooling. When the Gemini quota is exhausted, some modules fall back to mock data so the live demo stays usable on a cold-started free-tier dyno. The path to production is documented in TECHNICAL.md.

Feature № 06 · AI / Computer Vision Shipped · 2026 · TEJASKP
06

AI Photo Matcher — find your photos at a wedding by uploading one selfie.

A full-stack event-photography platform: photographers upload albums, faces are indexed by InsightFace, guests pay via Razorpay (or demo mode), upload one selfie, and get back only the photos they appear in. Matching is scoped to a single event for privacy.

Wedding and event photographers ship guests an album of 2,000+ photos. Guests want the 30 they’re actually in. Today they scroll. Photo Matcher cuts that to a single selfie upload — InsightFace embeds every face in the album at upload time, the guest’s selfie is embedded the same way, cosine similarity against just this event’s embeddings returns the personalised gallery.

i. Team of three, my slice

Three-person build at TEJASKP: a frontend engineer on the Next.js / Tailwind UI, an ML engineer on the InsightFace pipeline, and me on backend + integration — FastAPI service layer, PostgreSQL schema (events, photos, faces with coordinates & confidence, match sessions), JWT auth for photographers, multipart S3 upload handling, QR-code generation for event sharing, and the full Razorpay payment flow with a graceful demo-mode bypass when keys aren’t configured. The integration seam — making the AI pipeline run automatically after upload and exposing it cleanly through public endpoints — was mine.

ii. Architecture

iii. Two decisions worth defending

Index at upload, not at match. The expensive operation (InsightFace detection + 512-dim embedding) happens once per photo when the album is uploaded, not once per guest. Match time then collapses to a Postgres query + a tight cosine-similarity loop over only this event’s embeddings. A wedding with 2,000 photos and 200 guests does 2,000 detections, not 400,000.

Event-scoped matching as a privacy primitive. The query WHERE faces.event_id = :current_event isn’t a performance choice — it’s a policy boundary. A guest’s selfie can’t leak across events; the threshold (FACE_MATCH_THRESHOLD=0.45) and minimum face confidence (SELFIE_MIN_FACE_CONFIDENCE=0.55) are tunable, but the event scope is structural. The selfie validator also rejects any image that doesn’t contain exactly one face, which closes the “group photo as input” abuse vector.

The interesting backend problem wasn’t the face matching — it was making the AI pipeline behave like a synchronous experience while running asynchronously, and degrading the Razorpay path to a demo-mode bypass without changing any frontend code. — Integration notes

iv. Honest limits

Cosine similarity over InsightFace embeddings is a strong baseline but isn’t robust to large pose / lighting / age gaps; a guest who looks meaningfully different from their selfie won’t match. Threshold tuning trades recall for false positives, and at scale (10K+ photos per event) the per-event linear scan needs an ANN index (HNSW / FAISS-per-event) rather than a Postgres-side loop. Both are next-iteration concerns rather than design flaws.

Feature № 07 · AI / ML Shipped · 2026
07

WikiQA RAG — retrieval-augmented question answering, end-to-end.

A full retrieval-augmented QA pipeline over live Wikipedia: it splits articles into passages, retrieves the most relevant ones, re-ranks them for precision, guarantees a spread of sources, streams answers token-by-token from multiple LLMs, and grades its own faithfulness with an LLM judge.

Most RAG demos stop at “embed corpus, retrieve top-k, prompt LLM.” The interesting failures live one layer deeper — answer collapse to a single source article, retrieval that quietly degrades on multi-topic queries, and no honest measurement of faithfulness. WikiQA was the excuse to build the whole stack, including the evaluator.

i. The pipeline

Fetch 1–5 Wikipedia articles, strip markup, chunk into overlapping sentence windows via NLTK (1-sentence overlap, so context isn’t cut at chunk edges). Encode with all-MiniLM-L6-v2 → 384-dim L2-normalised vectors. FAISS IndexFlatIP for exact cosine, or a Pinecone serverless drop-in behind a VECTOR_STORE flag. The bi-encoder is fast but scores query and passage independently — a cross-encoder (ms-marco-MiniLM-L-6-v2) then re-scores the candidate set jointly. Cost: +500 ms latency, +20 points of Precision@5 (0.60 → 0.80; MRR 0.68 → 0.85 on 8 benchmark queries). Worth it for accuracy-critical use; toggleable for RAM-constrained boxes.

ii. Source diversity — the step that isn’t in the tutorials

The cross-encoder will happily fill all five top-k slots from the single most-relevant article and silently drop every other source. Great for precision, terrible for the LLM’s context breadth. The fix is not a penalty — it’s a guarantee. An O(n) post-rank step reserves one slot per unique fetched article before filling the remaining slots by cross-encoder score. Zero latency cost. Noticeable improvement on multi-topic queries (“Compare Python and Rust memory models”), which were the worst category before.

iii. Measuring faithfulness

Precision@k and MRR tell you whether the right passages were retrieved — not whether the LLM stayed faithful to them. So the generator’s LLM is asked, on its own output, “Given only these passages, is this answer supported? YES or NO.” Same pattern as Cohere’s RAG evaluator and Ragas. Not ground truth, but it catches the most common failure mode — plausible-sounding facts the model invented that aren’t in the retrieved Wikipedia text.

The honest signal isn’t “did the LLM say something plausible” — it’s can a second pass trace every claim back to a passage we actually retrieved. — Eval notes

iv. Production surface

FastAPI v4 backend · React 18 + Vite + Tailwind chat UI · SSE token streaming via /ask/stream for word-by-word output · multi-LLM dispatcher (Groq llama-3.3-70b default, OpenAI, Anthropic, local flan-t5-small) · JWT auth with per-user SQLite history · content-hash FAISS persistence (re-running the same query skips both re-encoding and re-fetching) · Prometheus metrics + Grafana dashboard via docker-compose · 46 pytest cases · Docker image + GitHub Actions CI/CD · tuned to fit a t2.micro (1 GB RAM + 2 GB swap).

Research № 01 · Formal Methods · Distributed Security Preprint · May 2026
08
Preprint · not peer-reviewed

When Local Correctness Is Not Enough — a TLA⁺-style proof that three RBAC invariants break under concurrency.

A formal specification and model checker for three security invariants that every individual service upholds, yet the composed system violates under specific cross-service orderings. Exhaustive state-space exploration; all three counterexamples reproduced. Zenodo doi:10.5281/zenodo.20686317.

While building a multi-role RBAC marketplace I hit a concurrent-order race that a per-service correctness argument said couldn’t happen. This paper is the formalisation of why it can. It models a five-service distributed topology — Identity, Order, Inventory, API Gateway, Notification — and states three invariants precisely, then lets a breadth-first checker enumerate every reachable state and test each invariant at each one. The point is honest and narrow: a property proven locally per service is not preserved under composition.

i. The three invariants

RPS (Role-Permission Separation) and APC (API Gateway Policy Consistency) both require a service’s policy cache to stay a subset of what the user’s current roles authorise — broken when a role is revoked or downgraded but the gateway cache sits stale inside its TTL window. TAR (Transaction Atomicity Under Revocation) requires that no order stays pending for a user whose token was revoked — broken when revocation is re-checked only at confirm-time, leaving a reachable (revoked ∧ pending) state. Each holds per-service; the composition does not.

ii. The correction that mattered

An earlier draft indexed the cache invariant by role label — cache[s][r] ⊆ permissions(r). That predicate is trivially true: permissions is a static map, so it can never be falsified. The authorisation decision a gateway actually makes is over a user’s effective permission set, which is what changes under role mutation. The spec and checker now both verify the user-indexed form cache[s][u] ⊆ permissions(roles[u]) — the version that can actually be violated, and is.

iii. What the numbers say

On the finite instantiation Users={u0,u1}, Roles={ADMIN,BUYER,FARMER}, the buggy spec reaches 188,160 distinct states — all three invariants violated — while the fixed spec collapses to 4,928 and stays clean. The defect doesn’t merely permit unsafe states; it enlarges the reachable space, because each independent stale-cache configuration is its own state. Under the fix the cache is a deterministic function of the role assignment, so that explosion disappears. One command reproduces the whole run.

TAR preservation requires revocation to cancel pending orders atomically, in the same transition step. The checker surfaced an exploitable intermediate state the original engineering argument never anticipated. — Headline finding

iv. Scope & honesty

This is forward-looking architectural analysis, not a forensic audit of running code, and the preprint is not peer-reviewed. The contribution is the invariant specification, the machine-checked violation traces, and a verified fix — at the architectural level. The deployed system that motivated it (AgriFuture) is a single Express server; the formal model targets the five-service topology that single server would grow into. I’d rather state that plainly than overclaim.
